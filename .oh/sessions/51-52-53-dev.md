# Session: 51-52-53-dev

## Aim
**Updated:** 2026-04-08

## Aim Statement

**Aim:** Integrators and local developers can create endeavors and reset local Open Horizons instances without guessing at hidden API rules or Docker state semantics.

**Current State:** Create requests can silently diverge from persisted state, valid type values are not authoritative from the integration surface, and local Docker reset behavior is easy to misread.

**Desired State:** The create surface is strict and truthful, the same surface exposes authoritative type vocabulary and field names, and the README makes reset outcomes predictable.

### Mechanism
**Change:** Tighten `POST /api/endeavors/create` around one explicit request contract, make `/api/about` the live integration-time source for create fields and type slugs, and document reset levels in README.

**Hypothesis:** If callers can discover the create contract where they integrate and the route either persists or rejects each field explicitly, hackathon-style friction drops without inventing parallel APIs.

**Assumptions:**
- `/api/about` is a coherent discoverability surface for integrators already using HTTP.
- Adding strict request validation will not break supported first-party callers once they use canonical field names.
- README is the right place for local reset semantics.

### Feedback
**Signal:** Real create requests can persist `description`, reject unknown fields like `parent_id`, point invalid-type callers to an authoritative vocabulary, and README clearly distinguishes routine cleanup from clean-slate reset.

**Timeframe:** Verified in this branch with focused API tests and a live API spot-check before merge.

### Guardrails
- Keep node type authority data-driven from `node_types`; do not re-hardcode primitive type names.
- Do not expand this batch into issue #54's custom-node-type write-path investigation.
- Preserve the graph's advisory hierarchy model; do not add DB-level type-parent enforcement.

## Problem Statement
**Updated:** 2026-04-08

**Current framing:** Three separate hackathon issues need fixes: create contract truthfulness (#51), type discoverability (#52), and Docker reset docs (#53).

**Reframed as:** Open Horizons' integrator-facing setup and create path is missing one truthful, authoritative contract surface, and local reset docs do not accurately communicate persistence boundaries.

**The shift:** #51 and #52 are the same API contract/discoverability problem at two points in the flow; #53 is docs-only but comes from the same first-run friction loop and can ship alongside the API fix without forcing unrelated architecture.

### Constraints
- **Hard:** No issue #54 work in this batch. Node types remain data from `node_types`. Hierarchy hints stay advisory.
- **Soft:** A docs-only issue would normally split, but here it is small and tied to the same hackathon onboarding story.

### What this framing enables
- A single PR that aligns runtime validation, integration discoverability, and onboarding docs.
- Tight scope around the create/about/README surfaces instead of a broad importer or hierarchy refactor.

### What this framing excludes
- Investigating the broader custom-node-type write path beyond what #51/#52 need.
- Reworking unrelated CRUD responses or adding a dedicated types endpoint unless the existing surfaces prove insufficient.

## Solution Space
**Updated:** 2026-04-08

## Solution Space Analysis

**Problem:** Integrators cannot reliably infer what `POST /api/endeavors/create` accepts or whether accepted input was actually persisted, and local reset docs hide named-volume persistence.

**Key Constraint:** Keep type authority live and data-driven; do not hardcode create-time vocabulary into docs or route-local conditionals.

### Candidates Considered

| Option | Level | Approach | Trade-off |
|--------|-------|----------|-----------|
| A | Local Optimum | Patch create route to persist `description`, reject unknown fields, and update README only | Leaves discoverability split across ad hoc surfaces |
| B | Reframe | Make `/api/about` the authoritative live create contract surface, point validation errors to it, and keep create response narrow but explicit | Requires touching both route and introspection output |
| C | Redesign | Add a dedicated `/api/types` or `/api/contracts/create` endpoint and refactor callers around it | More surface area than needed for this batch |
| D | Split batch | Land #51/#52 in one PR and #53 separately | Cleaner thematic separation, higher coordination overhead for one small docs fix |

### Evaluation

**Option A: Patch create only**
- Solves stated problem: Partially
- Implementation cost: Low
- Maintenance burden: Medium
- Second-order effects: Runtime improves, but integrators still rely on docs or trial/error for type discovery

**Option B: Tighten create + elevate `/api/about`**
- Solves stated problem: Yes
- Implementation cost: Medium
- Maintenance burden: Low
- Second-order effects: One runtime surface explains fields and valid types; docs can point at live truth

**Option C: New endpoint**
- Solves stated problem: Yes
- Implementation cost: Medium
- Maintenance burden: Medium
- Second-order effects: Another integration surface to keep aligned with `/api/about` and docs

**Option D: Split #53**
- Solves stated problem: Yes, in two steps
- Implementation cost: Medium
- Maintenance burden: Medium
- Second-order effects: More process overhead for a tiny README fix tied to the same feedback batch

### Recommendation

**Selected:** Option B, with #53 kept in the same PR
**Level:** Reframe

**Rationale:** The user problem is not just one dropped field; it is the absence of a single truthful contract surface. Tightening create request validation and making `/api/about` the live authority covers both #51 and #52 without adding a new endpoint. #53 stays coherent because it is small, sourced from the same hackathon first-run friction, and touches only README onboarding.

**Accepted trade-offs:**
- `/api/about` becomes more integration-focused than before.
- The create response stays intentionally narrow if request truthfulness and discoverability are explicit.

### Implementation Notes
- Extend the shared create request contract to include canonical `description` and reject unknown keys.
- Keep node type validation/live vocabulary DB-backed in the route and `/api/about`.
- Update first-party create flows to use the supported `description` field directly where they currently workaround the route.
- Document Docker reset levels in README with explicit named-volume semantics.

## Dissent
**Updated:** 2026-04-08

## Dissent Report

**Decision under review:** Batch issues #51, #52, and #53 into one PR centered on the create contract and onboarding discoverability.
**Stakes:** User-reported first-run friction; batching too much could blur review, splitting too much could leave docs/runtime inconsistent.
**Confidence before dissent:** MEDIUM

### Steel-Man Position
One batch is reasonable because #51 and #52 are the same create-surface problem and #53 is a small README clarification from the same hackathon feedback loop. Reviewers can evaluate the branch as “make first-run integration/setup truthful.”

### Contrary Evidence
1. #53 is docs-only and could be reviewed faster on its own.
2. `/api/about` might not be the integration surface people naturally hit; changing it could miss the actual failure path.
3. Tightening request validation can break hidden callers that relied on silently ignored fields.

### Pre-Mortem Scenarios
1. **Batch sprawl:** Reviewers struggle to reason about API and Docker docs together, slowing merge.
2. **Wrong discoverability surface:** `/api/about` changes land but callers still do not find valid types where errors occur.
3. **Strictness regression:** Unknown-field rejection breaks a first-party caller that was unknowingly off-contract.

### Hidden Assumptions

| Assumption | Evidence | Risk if Wrong | Test |
|------------|----------|---------------|------|
| `/api/about` is acceptable as the authoritative list | Existing public route already summarizes product/API capabilities | Discoverability still weak | Add tests plus live spot-check of invalid-type errors pointing there |
| First-party callers already use canonical field names | Current UI sends `parentId` and can be updated to send `description` | Strict validation breaks internal flows | Run focused API tests and exercise first-party create path behavior via tests/code review |
| #53 is small enough not to obscure review | README-only change, same feedback source | PR loses coherence | Keep PR title/body explicit about hackathon friction batch and limit README diff |

### Decision

**Recommendation:** ADJUST

**Reasoning:** Batch the issues, but keep the implementation tightly centered on the create/about/README surfaces and explicitly exclude #54. Use `/api/about` only if invalid-type errors also point callers there; otherwise the discoverability story is incomplete.

**If ADJUST:**
- Do not broaden into importer or custom-node-type write-path work.
- Keep the create response narrow unless response expansion becomes necessary for truthfulness.
- Make the README reset section visibly outcome-based (`restart`, `rebuild`, `clean slate`).

**Confidence after dissent:** HIGH

**Create ADR?** No - this is a focused product-surface choice, not a one-way architectural door.
